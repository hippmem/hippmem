# 容量规划

> 你需要多少条记忆？10 万～100 万条对你有何意义？本文帮你评估。

---

## 一条记忆是什么？

HIPPMEM 中的一条记忆 = **一个原子事实**。不是文档、不是对话日志、不是段落。例如：

- 「比起 Python，我更倾向用 Rust 做后端」（一条偏好）
- 「决定用 redb，因为纯 Rust 无 C 依赖」（一条决策）
- 「引擎支持 14 种关联边类型」（一条知识点）

每次 `engine.write()` 调用产生一条记忆。HIPPMEM **不做自动分块**——调用方负责将对话或文档拆解为值得记住的原子事实。

---

## 具体示例：100 条记忆

`seed_100` 测试数据（用于测试和示例）覆盖了**一个开发者 + 一个项目**的完整知识画像：

| 类别 | 数量 | 示例内容 |
|---|---|---|
| 个人履历 | 10 | 姓名、角色、工作年限、语言能力 |
| 偏好 | 15 | 语言偏好、工具偏好、工作习惯 |
| 技术决策 | 20 | 为什么选 redb、为什么用 SimHash、为什么 bincode 替代 JSON |
| 项目知识 | 20 | crate 结构、算法参数、架构选型 |
| 任务状态 | 15 | 已完成 CI/CD、已修复 bug、进行中特性 |
| 观察记录 | 10 | 基准测试结果、外部工具评测 |
| 修正与反思 | 10 | 「之前用 X，后来换成 Y，因为 Z」 |

**100 条记忆 = 数月协作后你对一个人及其工作的全部了解。**

---

## AI Agent 每天产生多少条记忆？

取决于使用强度。以下估算假设 Agent 只写入**有意义的新事实**——并非每轮对话都产生记忆。

| 使用强度 | 条/天 | 典型场景 |
|---|---|---|
| **轻度个人** | 10～20 | 偶尔记笔记、偏好、提醒；每天几次对话 |
| **日常专业** | 30～80 | 每日编程助手、研究伙伴、知识管理 |
| **重度专业** | 100～300 | 深度研究、文档拆解、持续知识提取 |
| **持续 Agent 循环** | 500～1,000 | Agent 7×24 高吞吐不间断运行 |

参考：单轮对话通常产生 0～2 条记忆。一次 50 轮的工作 session 可能产生 20～60 条。瓶颈不在引擎，而在于交互中产生了多少**新的、独特的、值得记住的事实**。

---

## 能用多久？

| 使用强度 | 10 万条 | 100 万条 |
|---|---|---|
| 轻度个人（~15/天） | **~18 年** | **~180 年** |
| 日常专业（~50/天） | **~5.5 年** | **~55 年** |
| 重度专业（~200/天） | **~1.4 年** | **~14 年** |
| 持续循环（~500/天） | ~6 月 | **~5.5 年** |
| 极端吞吐（~1,000/天） | ~3 月 | **~2.7 年** |

**对绝大多数用户，10 万条代表数年的使用，100 万条远超个人终身用量。**只有持续 7×24 高吞吐率的 Agent 循环才会在数月内填满 10 万条——即使如此，100 万条也能撑数年。

---

## 这是硬上限吗？

**不是。**10 万～100 万是 HIPPMEM 经过测试、性能舒适的区间。超过 100 万后系统继续工作——性能是渐进退化，而非突然崩溃：

- **redb**（主存储）：基于 B-tree，随规模渐进退化，瓶颈在 I/O
- **Tantivy**（全文索引）：原生设计支持百万级文档
- **向量索引**：性能取决于后端和可用 RAM

**提升硬件配置可扩展舒适区间。**更多 RAM、更快的 NVMe SSD、更多 CPU 核心都能推高实践上限。笔记本上 100 万条可能是上界；32 GB RAM + NVMe 服务器上 100 万条只是中位。

---

## 未来可扩展性

10 万～100 万反映的是当前单机单文件架构下的舒适区间。以下技术路径可在未来版本中将上限提升一个数量级（1000 万+）：

| 方向 | 手段 | 预期增益 |
|---|---|---|
| **分区存储** | 按时间或主题拆分为多个 redb 文件，降低单文件 B-tree 深度 | 3～5× |
| **分级索引** | 热数据全索引，冷数据轻量索引 | 2～3× |
| **记忆压缩** | 合并冗余或已被取代的记忆，自动整合旧事实 | 2～5×（有效容量） |
| **ANN 向量索引** | 近似最近邻替代暴力搜索，支撑大规模语义检索 | 解锁 1000 万+ |
| **格式优化** | 更紧凑的序列化、关联键增量编码 | 1.5～2× |
| **硬件红利** | NVMe 带宽持续增长，RAM 性价比逐年提升 | 每代硬件 ~2× |

分区存储 + 分级索引 + 记忆压缩三者结合，即可将舒适区间推至 **1000 万条以上**——足以支撑一个小型企业运行多个 Agent 十年之久。这些都不需要架构重写，是现有设计的增量演进。

---

## 总结

| 担心 | 现实 |
|---|---|
| 「会不会几个月就满了？」 | 只有极端 7×24 高吞吐场景才会。日常使用以年计。 |
| 「超过 100 万怎么办？」 | 100 万是软性设计目标，非硬上限。更好硬件 → 更高天花板；规划中的改进目标 1000 万+。 |
| 「我该为这个担心吗？」 | 个人或小团队用户基本不会触达上限。高吞吐生产环境建议按 100 万规划并监控。 |
